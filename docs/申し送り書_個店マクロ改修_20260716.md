# 申し送り書 個店マクロ改修（フェイス陳列数の計算順序変更：店型2加算を上下限クランプ前へ）

作成日: 2026-07-16

依頼者と確定した修正要件の記録です。実装は Codex が本書のみを頼りに行える粒度で記載しています。
仕様の正本は `要件定義書_在庫自動調整マクロ.md` / `設計書_在庫自動調整マクロ.md` であり、
本改修の反映時にそちらも更新してください（更新対象箇所は §5 に列挙）。

## 1. 背景

### 1.1 依頼内容（原文の趣旨）

フェイス陳列数に入れる数値を計算する時の計算の順番を変更し、思惑に沿った調整ができるようにする。
現行は「下限値を算出（適用）した後に、店型2陳列数(本部推奨)があればそれを加算する」順番のため、
**意図せず在庫量が跳ね上がることがある**。そこで、

> 倍率計算 → 手持週数乗算 → 店型2を加算 → 上下限値でクランプ

の順番になるよう変更する。

※依頼者の言う「フェイス２」は、シート上の「店型２陳列数(本部推奨)」行
（本部推奨フェイス2、`planRow + FACE_HQ_2_OFFSET` = `planRow + 13`）を指す。

### 1.2 用語対応表（本書内で使う用語とコードの対応）

| 本書の用語 | コード上の対応 |
| --- | --- |
| フェイス陳列数 | マクロが書き込む陳列数行（`stockRow`）の値 |
| 店型2（店型2陳列数(本部推奨)） | 本部推奨フェイス2行 `planRow + FACE_HQ_2_OFFSET`（= +13）。加算値の取得は `GetStoreType2FaceAddition` |
| 本部推奨フェイス1 | `planRow + FACE_HQ_1_OFFSET`（= +12）。参照関数は `HqFaceBase` |

| 運用側下限値 | 実行時に決まる最下限。`GetPriorityFaceQty` / `RunStockControlCore` の引数 `minVal`（自動調整は `DEFAULT_MIN_VAL` = 5、手動系は操作者入力）。設計書の「指定下限値」「運用側最下限」と同義。**§1.3 手順1 に出てくる `minVal:=0` は `GetBestFaceQty` 内部呼び出しの引数指定であり、運用側下限値とは別物** |

| 実効下限 | `Max(シート下限の切り上げ値, relationFloor)`。本改修後のクランプ下限（R1 手順3） |

### 1.3 現行の計算順序と問題点

フェイス陳列数の算出は `modCalculations.bas` の `GetPriorityFaceQty`（:226-264）に集約されている。
現行の順序:

1. `GetBestFaceQty(販売計画, シート下限, シート上限, minVal:=0, targetWeeks)`（:242）—
   手持週数×販売計画（週数数量）を整数化するが、**この時点でシート下限・上限にクランプ**する
   （週数数量 ≤ シート下限なら下限値を返す。:198-201）
2. `relationFloor`（本部推奨フェイス1 > 運用側下限値なら運用側下限値、
   0 < フェイス1 ≤ 運用側下限値なら Ceil(フェイス1)、それ以外＝フェイス1 ≤ 0
   （空白・非数値は 0 扱い）なら 0。:244-250）と `Max` を取る（:252-256）—
   **ここでも下限へ持ち上げ**
3. `addStoreType2 = True` なら **店型2加算**（`GetStoreType2FaceAddition`。:259-261）
4. `ClampFaceValueToSheetLimits`（:263）— シート上下限で最終クランプ
   （上限側は効くが、下限側は手順1〜2で既に持ち上がっているため実質効かない）

つまり **下限適用（手順1〜2）→ 店型2加算（手順3）** の順のため、
週数数量が実効下限より小さい SKU では「下限に持ち上げた値」の上に店型2が丸ごと乗る。

問題例（シート下限10、運用側下限値5、店型2 = 5、目標手持週数2.0）:

| 販売計画 | 週数数量 | 現行結果 | 依頼者の期待 |
| --- | --- | --- | --- |
| 3 | 6 | Max(10, 5) + 5 = **15** | 6 + 5 = 11 → クランプ → **11** |
| 1 | 2 | Max(10, 5) + 5 = **15** | 2 + 5 = 7 → クランプ → **10** |

需要（週数数量）が小さいほど下限持ち上げ幅が大きく、そこへ店型2が加わって在庫が跳ね上がる。

## 2. 確定要件

### R1. フェイス陳列数の計算順序を「加算 → クランプ」へ変更

新しい順序（`GetPriorityFaceQty` 内部）:

1. **倍率計算・手持週数乗算**（既存処理のまま変更なし）:
   販売計画は従来どおり倍率計算（`CalcRatioByTermAverage` → `CalcAutoPlanValue`、
   販売計画上下限クランプ含む）で先に確定・書込済み。フェイス計算は
   `週数数量raw = 目標手持週数 × 販売計画` を整数化する。
   **整数化は現行 `GetBestFaceQty` の候補選択ロジック（modCalculations.bas:208-223）を
   そのまま流用し、上下限クランプだけを行わない**:
   - 販売計画 ≤ 0 の場合は週数数量 = 0（現行 :190-193 に対応）
   - `targetWeeks ≤ 0` は 2.0 とみなす（現行 :195 維持）
   - 候補 `c1 = CLng(Fix(週数数量raw))`、`c2 = c1 + 1` のうち、
     **`lowDiff <= highDiff + EPS` なら c1、それ以外は c2**
     （`lowDiff = Abs(c1 ÷ 販売計画 − targetWeeks)`、highDiff も同様。
     EPS = 1e-7、modCommon.bas:23。現行 :216-219 の比較式をそのまま移植すること。
     「同値なら c1」ではなく **c2 が EPS 以内で近い場合も c1** という c1 寄りの判定である点に注意）
   - 現行 :211-214 の候補の [lo, hi] クランプは行わない
     （現行でも前段の早期リターンにより実質恒等であることを検証済み）
   - **Long あふれ防止**: 整数化の前に `週数数量raw > MAX_COUNT` なら `週数数量 = MAX_COUNT`
     として候補選択を省略する（最終上限は常に `MAX_COUNT` 以下のため結果は変わらない。
     現行は :203-206 の早期リターンが同じ役割を担っていた）
2. **店型2加算**: `addStoreType2 = True` の場合のみ
   `週数数量 + GetStoreType2FaceAddition(...)`（空白・数値以外・0以下は加算0。現行関数のまま）
3. **上下限クランプ**（加算後に一括適用）:
   - 下限 = **実効下限** = `Max(CeilToLong(シート下限), relationFloor)`
     （`relationFloor` の決定ルールは現行 :244-250 のまま変更しない）
   - 上限 = `GetFaceUpperLimit`（空白・0 は `MAX_COUNT`。現行のまま）
   - `ClampLong(加算後値, 実効下限, 上限)`（`ClampLong` は上限 < 下限のとき下限を優先する。現行仕様のまま）
4. **最終シート上下限クランプの維持**: 手順3の後にも従来どおり
   `ClampFaceValueToSheetLimits` を通す（前回改修 R1 で確定した
   「最終書込値のシート上下限クランプは全経路で必ず維持」のポリシー。手順3の実効下限が
   シート下限を含むため冗長になるが、MDシステム取込エラー防止の安全網として残す）。

参考擬似コード（新 `GetPriorityFaceQty` の中核。relationFloor 決定・変数取得は現行のまま）:

```vb
' --- 手順1: 週数数量の整数化（クランプなし） ---
If targetWeeks <= 0# Then targetWeeks = 2#
If planVal <= 0# Then
    weeksQty = 0
Else
    rawQty = targetWeeks * planVal
    If rawQty > CDbl(MAX_COUNT) Then
        weeksQty = MAX_COUNT            ' Long あふれ防止。最終上限 <= MAX_COUNT のため結果不変
    Else
        c1 = CLng(Fix(rawQty)): c2 = c1 + 1
        lowDiff = Abs((CDbl(c1) / planVal) - targetWeeks)
        highDiff = Abs((CDbl(c2) / planVal) - targetWeeks)
        If lowDiff <= (highDiff + EPS) Then weeksQty = c1 Else weeksQty = c2
    End If
End If

' --- 手順2: 店型2加算 ---
If addStoreType2 Then weeksQty = weeksQty + GetStoreType2FaceAddition(ws, planRow, colNo)

' --- 手順3: 実効下限・上限でクランプ ---
lowerBound = CeilToLong(lowerFace)      ' lowerFace = シート下限（planRow + DISP_LOWER_OFFSET）
If relationFloor > lowerBound Then lowerBound = relationFloor
candidateVal = ClampLong(weeksQty, CDbl(lowerBound), CDbl(GetFaceUpperLimit(ws, planRow, colNo)))

' --- 手順4: 最終シート上下限クランプ（安全網、現行維持） ---
GetPriorityFaceQty = ClampFaceValueToSheetLimits(candidateVal, ws, planRow, colNo)
```

### R2. 挙動が変わるケース・変わらないケース（回帰保証）

変わる／変わらないの境界は **実効下限（= Max(シート下限の切り上げ値, relationFloor)）** で判定する。
relationFloor 単独でもシート下限単独でもない点に注意。

**変わるケース**（すべて依頼者の意図どおりの変化）:

- 店型2 > 0 かつ **週数数量 < 実効下限** の週:
  従来「実効下限 + 店型2」→ 今後「Max(週数数量 + 店型2, 実効下限)」（上限内の場合）。
  店型2は**実効下限に部分的または完全に吸収され得る**（「常に +N 上乗せ」ではなくなる）。
  これが本改修の目的そのもの。
- 販売計画 ≤ 0（倍率0等）かつ 店型2 > 0 の週:
  従来「実効下限 + 店型2」→ 今後「Clamp(店型2, 実効下限, 上限)」。
- 本部推奨優先の戻し（`AdjustFaceForLowHqIfNeeded`、フェイス1 ≤ 運用側下限値）経由も
  同じ `GetPriorityFaceQty` を通るため、同様に変わる（relationFloor = Ceil(フェイス1) のケース）。

**変わらないケース**（数学的に同値であることを境界値含め検証済み。回帰確認ポイント）:

- **店型2が空白・0以下の週**: 加算0のため、旧「クランプ→Max→+0→クランプ」と
  新「+0→クランプ」は全ケースで同じ値になる。
- **HB食品用調整**（`addStoreType2 = False`）: 上と同じ理由で全 SKU 結果不変。
- **週数数量が実効下限以上の週**: 従来も下限持ち上げが発生しないため、
  「週数数量 + 店型2 → 上限クランプ」で同じ値になる。
- **上限側**: 従来も加算後の最終クランプで上限に収めていたため、変化なし。
- **店型2の役割**: 引き続き最終加算専用。対象判定（`ShouldAdjustByHqFace`）・
  下限側の寄せ先（`HqFaceBase` / relationFloor）には従来どおり使わない。
  **販売予定なし判定は従来どおりフェイス1と店型2の両方で行う**
  （`IsNoSalesPlanWeek` / `HasAnySalesPlanTargetWeek` は変更しない。
  HB食品用はフェイス1のみで判定、これも現行維持）。
- **販売予定なし週のスキップ・新商品スキップ・意思入れ済みセルの補正
  （`ClampExistingFaceIfNeeded`）・フェイス1=0で店型2のみの週の扱い
  （計算せず既存値のシート上下限補正のみ）**: すべて現行維持。

数値例（シート下限10、シート上限40、運用側下限値5、フェイス1 > 5 → relationFloor = 5、
実効下限 = Max(10, 5) = 10、店型2 = 5、手持週数2.0）:

| 販売計画 | 週数数量 | 実効下限との関係 | 現行 | 改修後 | 変化 |
| --- | --- | --- | --- | --- | --- |
| 3 | 6 | 6 < 10 | 15 | 11 | 跳ね上がり解消 |
| 1 | 2 | 2 < 10 | 15 | 10 | 店型2が実効下限に吸収 |
| 8 | 16 | 16 ≥ 10 | 21 | 21 | 変化なし |
| 20 | 40 | 40 ≥ 10 | 40 | 40 | 変化なし（上限で停止） |

## 3. 実装ガイド

### 3.1 変更箇所

- 変更は `src/vba/modCalculations.bas` の `GetPriorityFaceQty`（およびその下請けの
  `GetBestFaceQty`）に閉じる。**`GetPriorityFaceQty` のシグネチャは変えない**
  （呼び出し元2箇所を無修正で済ませるため）。
- `GetBestFaceQty` の呼び出し元は `GetPriorityFaceQty`（modCalculations.bas:242）のみのため、
  シグネチャ変更・削除は自由。次のいずれかとし、関数責務のコメントは実態に合わせて更新すること:
  - 「クランプなしの週数数量整数化」関数に作り替える（未使用になる引数は削除を推奨）
  - 整数化部分を新関数へ切り出す。この場合、旧 `GetBestFaceQty` は呼び出しゼロの死コードに
    なるため**削除してよい**（残す場合は、旧クランプ仕様が誤って再利用されないよう
    コメントで注意書きを付けること）
- `GetPriorityFaceQty` の呼び出し元は次の2箇所で、いずれも修正不要（内部変更のみで反映される）:
  - `modStockEngine.bas:143`（通常のフェイス計算。自動調整・手動調整・HB食品用調整の共通経路）
  - `modCalculations.bas:345`（`AdjustFaceForLowHqIfNeeded` = 本部推奨優先の戻し）
- `ClampFaceValueToCurrentLimits`（modCalculations.bas:266）は現在どこからも
  呼ばれていない。本改修では触らなくてよい（削除するかは実装者判断。残す場合は
  旧順序のロジックが誤って再利用されないようコメントで注意書きを付けること）。

## 4. 実装時の要件外決定事項

| 項目 | 決定 |
| --- | --- |
| 整数化と店型2加算の順序 | 週数数量を先に現行ルール（EPS 許容付き最近接候補選択）で整数化し、その後に店型2（`CeilToLong` 済みの整数）を加算する。加算後の再丸めは不要 |
| relationFloor の適用位置 | 「下限へ持ち上げてから加算」をやめ、加算後クランプの下限の一部（実効下限 = Max(シート下限, relationFloor)）として適用する。決定ルール自体（フェイス1と運用側下限値の大小比較）は変更しない |

| 最終シート上下限クランプ | 冗長でも全経路で維持（前回改修 R1 のポリシー踏襲） |
| 販売計画算出側 | 一切変更しない（倍率計算・販売計画上下限クランプは現行のまま） |

## 6. 動作確認チェックリスト（依頼者実施）

※「実効下限」= Max(シート下限, 下限側の寄せ先)。§2 R2 の定義と同じ。

- 店型2 > 0 かつ 週数数量が実効下限未満の SKU: 従来「実効下限 + 店型2」だった値が
  「Max(週数数量 + 店型2, 実効下限)」に下がる（§2 の数値例で確認。
  ただし週数数量 + 店型2 がシート上限以上の SKU は改修前後とも上限値で変わらない）
- 店型2 > 0 かつ 週数数量が実効下限以上の SKU: 改修前後で値が変わらない
- 店型2が空白・0 の SKU: 改修前後で値が変わらない
- HB食品用調整: 改修前後で全 SKU の値が変わらない
- 本部推奨優先の戻し対象（フェイス1 ≤ 運用側下限値）の SKU でも新順序で計算される。
  期待値の例: フェイス1 = 4、運用側下限値5、シート下限 空白(0)、週数数量2、店型2 = 3 →
  relationFloor = Ceil(4) = 4、実効下限 = 4 → 旧: Max(2, 4) + 3 = **7** → 新: Max(2 + 3, 4) = **5**
- 週数数量 + 店型2 が上限を超える SKU: 書込値が上限で止まる
- 全パターンで書込値がシート上のフェイス陳列数下限・上限内に収まる
- 販売予定なし判定（フェイス1・店型2両方0で週スキップ。HB食品用はフェイス1のみ）・
  新商品スキップ・意思入れ済みセルの上下限補正・フェイス1=0で店型2のみの週の扱いが従来どおり動く

  以下は依頼者が記入。
  「本部推奨優先の戻し対象（フェイス1 ≤ 運用側下限値）の SKU 」は、店型２＝０ならば、計算不要。
  また、店型２の数値はそのままの数量が加算されるか、半分の数値で加算されるのか、現在のロジックではどちらが採用されているのか開発する前に教えてください。
