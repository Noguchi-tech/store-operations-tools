# 開発ガイド 在庫自動調整マクロ

作成日: 2026-07-03

このドキュメントは、プロジェクトに初めて参加するエンジニアが
**保守（コード変更）と配布（アドイン更新）** を一人で回せるようになるための実務手順書です。
仕様そのものは `要件定義書_在庫自動調整マクロ.md` と `設計書_在庫自動調整マクロ.md` を正本とします。

## 1. 全体像

```text
src/vba/*.bas（コードの正本）
      │  tools/sync_vba_to_books.ps1 で取り込み
      ▼
src/src.xlsm（編集元ブック） と releases/在庫自動調整.xlam（配布物）
      │  releases/在庫自動調整.xlam を利用者へ渡す
      ▼
利用者の Excel（アドインとして常駐、リボン「便利ツール」タブ）
```

- VBA コードは `src/vba/*.bas` をテキストとして管理し、Git で差分を追います。
- ブック2つ（`src.xlsm` / `在庫自動調整.xlam`）の中の VBA は、`.bas` から取り込んだコピーです。
  **必ず `.bas` を先に編集し、スクリプトで両ブックへ反映してください。**
  VBE 上で直接編集すると、Git 上のコードと実物がずれて事故のもとになります。
- リボンの定義は `src/ribbon/customUI14.xml` で管理します。ブック内の
  `customUI/customUI14.xml` という zip エントリ（xlsm / xlam は実体が zip ファイル）に格納されます。

## 2. 開発環境の準備

| 必要なもの | 補足 |
| --- | --- |
| Windows + デスクトップ版 Excel | COM 自動化を使うため必須（Microsoft 365 の Excel 16.x で動作確認） |
| PowerShell | Windows 標準の 5.1 でも PowerShell 7 (`pwsh`) でも可 |
| Git | このリポジトリの管理 |

### 2.1 VBE 自動化の許可（初回のみ）

同期スクリプトが VBA プロジェクトへアクセスできるよう、Excel 側で以下を有効にします。

- Excel → ファイル → オプション → トラストセンター → トラストセンターの設定
  → マクロの設定 → **「VBA プロジェクト オブジェクト モデルへのアクセスを信頼する」にチェック**

（レジストリでは `HKCU\Software\Microsoft\Office\16.0\Excel\Security` の `AccessVBOM=1`）

## 3. 文字コードと改行の約束（重要）

| 対象 | 約束 |
| --- | --- |
| `src/vba/*.bas` | **CP932（Shift-JIS）+ CRLF**。UTF-8 で保存し直さないこと |
| Markdown / XML | UTF-8 |
| `tools/*.ps1` | **UTF-8（BOM 付き）**。BOM が無いと Windows PowerShell 5.1 が日本語を誤読する |

- VBE は `.bas` を ANSI（日本語環境では CP932）として取り込むため、UTF-8 のまま取り込むと
  日本語のコメント・文字列がすべて文字化けします。
- `.gitattributes` で `*.bas text eol=crlf` を指定済みのため、通常のチェックアウトでは
  自動的に CRLF になります。エディタ側で LF に変えないよう注意してください。
- 各 `.bas` の先頭行 `Attribute VB_Name = "モジュール名"` は VBE が取り込み時に使う正式名です。
  消したり変えたりすると `Module1` などの仮名で取り込まれます。
- CP932 に無い文字（例: 一部の環境依存文字）はコード内に書けません。どうしても必要な場合は
  既存コードにある `ChrW$(&H....)` 方式で Unicode コードポイントを直接指定します。

## 4. コード変更から反映まで

### 4.1 日常の流れ

```powershell
# 1. src/vba/*.bas を編集する

# 2. Excel をすべて閉じてから、同期スクリプトを実行する
pwsh -File tools/sync_vba_to_books.ps1

# 3. PASS を確認して、コミット・push する
git add -A
git commit -m "変更内容"
git push
```

### 4.2 tools/sync_vba_to_books.ps1 がやること

1. EXCEL プロセスが残っていないか確認（残っていれば中止）
2. `src/src.xlsm` と `releases/在庫自動調整.xlam` を `%TEMP%\koten-macro-backup\` へバックアップ
3. リボン XML（`src/ribbon/customUI14.xml`）を zip 差し替えで両ブックへ反映
   （Excel で開く前に行う。Excel 保存後も customUI パートは保持される）
4. Excel を非表示で起動し、両ブックの `mod*` 標準モジュールを `src/vba` と同期
   （同名モジュールを入替え、`src/vba` に無い `mod*` の残留モジュールは削除）
5. VBE の「コンパイル」コマンド（CommandBar コントロール ID 578）でプロジェクト全体をコンパイル
   - コンパイルエラー時はモーダルダイアログで固まるため、**ウォッチドッグが一定時間後に
     EXCEL を強制終了**してスクリプトはエラー扱いで停止します
6. 一時モジュール経由で `DebugCompileSmokeTest` を実行し、True が返ることを確認
   （`Option Private Module` の関数のため、同一プロジェクト内から呼び出す）
7. 取り込んだモジュールをエクスポートして `src/vba` と diff、リボンを zip から抽出して
   `src/ribbon/customUI14.xml` と byte 比較 → すべて一致で PASS

主なオプション:

```powershell
# リボンは差し替えず、モジュールだけ反映する
pwsh -File tools/sync_vba_to_books.ps1 -SkipRibbon

# コンパイルのウォッチドッグ時間を変更する（既定 90 秒）
pwsh -File tools/sync_vba_to_books.ps1 -CompileTimeoutSec 180
```

### 4.3 スクリプトが使えない場合の手動反映

1. 対象ブックを Excel で開き、Alt+F11 で VBE を開く
2. プロジェクトエクスプローラーで既存の `mod*` モジュールを右クリック → 解放（エクスポートは不要）
3. ファイル → ファイルのインポートで `src/vba/*.bas` を全て取り込む
4. デバッグ → VBAProject のコンパイル でエラーが無いことを確認
5. イミディエイトウィンドウ（Ctrl+G）で `?DebugCompileSmokeTest()` を実行し `True` を確認
6. 上書き保存
7. リボンを変更した場合は、ブックを閉じた状態で zip として開き
   `customUI/customUI14.xml` を差し替える（7-Zip などを使用）

## 5. 検証

| 確認項目 | 方法 |
| --- | --- |
| コンパイルが通る | 同期スクリプトの PASS 表示、または VBE のコンパイルメニュー |
| smoke テスト | `DebugCompileSmokeTest()` が `True` |
| debug_log 出力 | イミディエイトから `デバッグログ動作確認` を実行 → ActiveWorkbook に `debug_log` シートが作られる |
| 動作確認の観点 | 設計書の「検証観点」章を参照 |

## 6. 配布とインストール

配布物は `releases/在庫自動調整.xlam` の1ファイルだけです（現在はチャット添付で渡す運用）。

### 6.1 利用者側の初回インストール

1. `在庫自動調整.xlam` を利用者PCの `%APPDATA%\Microsoft\AddIns\` へコピー
2. Excel → ファイル → オプション → アドイン → 管理「Excel アドイン」→ 設定
3. 「在庫自動調整」にチェック → OK
4. リボンに「便利ツール」タブが出れば完了

### 6.2 更新（差し替え）

1. 利用者の Excel をすべて終了する
2. 同じ場所の `在庫自動調整.xlam` を新しいファイルで上書き
3. Excel を起動して「便利ツール」タブの動作を確認

補足: アドイン内の VeryHidden シート `setting` は、以前「AM維持フラグ履歴保存先」の
前回設定の保存に使っていましたが、履歴ブックの保存先が Downloads 固定になったため現在は未使用です
（`modSettings` は汎用基盤として残しています）。

## 7. トラブルシューティング

| 症状 | 原因と対処 |
| --- | --- |
| 取り込んだモジュール名が `Module1` になる | `.bas` 先頭の `Attribute VB_Name` が無い。行を復元して取り込み直す |
| 日本語コメントが文字化けする | `.bas` が UTF-8 で保存されている。CP932 へ戻す（Git で復元が確実） |
| 同期スクリプトが「EXCEL が起動中」で止まる | タスクマネージャーで EXCEL.EXE を終了してから再実行 |
| コンパイル段階でタイムアウトする | コンパイルエラーの可能性が高い。VBE で対象ブックを開き、デバッグ → コンパイルでエラー箇所を特定 |
| `VBProject へのアクセスが拒否されました` | 2.1 の「VBA プロジェクト オブジェクト モデルへのアクセスを信頼する」が未設定 |
| リボンが表示されない | zip 内 `customUI/customUI14.xml` の有無を確認。XML の構文エラーでもタブごと消える |
| 同期スクリプトが「大文字小文字だけが異なる」と警告する | VBA は大文字小文字を区別せず、`Dim value As Long` のような小文字の変数宣言を一度でも取り込むと、ブック側の識別子テーブルが全モジュールの `.Value` を `.value` へ塗り替える（エクスポート結果が源泉とズレる）。プロパティ名と同名の変数（`value` など）を宣言しないこと。既に汚れたブックは、`Dim Value As Long` を含む一時モジュールを取込→削除→保存すると塗り直せる |
| 実行時エラーの調査 | 対象ブックの `debug_log` シート（自動作成）に処理名・段階・行・列・エラー内容が残る |

## 8. リポジトリ運用

- 既定ブランチは `main`。リモートは GitHub（`origin`）
- コミット前に `git status` で意図しないバイナリ差分（ブック2つ）が無いか確認
- ブックへ反映した時は、`.bas` の変更とブックのバイナリ差分を同じコミットに含めると
  「どのコード変更がどのブックに入ったか」が履歴で追いやすくなります
